## 1. Summary and locked decisions (owner)

Implement the v2 prototype as THE app shell — full port, staged:
1. **Scope:** tokens + shell + practice screen + zoomed live-camera fretboard pane +
   setup wizard reskin, one branch.
2. **Zoom pane is HYBRID:** calibrated → live cropped video of the real fretboard with
   dots re-projected; uncalibrated/camera-off → the existing schematic `FretboardStrip`.
3. **Explore entry:** compact `Practice | Explore` segmented toggle next to the lesson
   picker in the top bar (existing mode semantics unchanged).
4. **Theming: dark AND light now.** Dark is the prototype; light is a derived palette
   (owner tunes by eye afterward). The app already uses `data-theme` on
   `document.documentElement` — keep that mechanism, but v2 DEFAULTS TO DARK
   (`data-theme="dark"` set on first load), toggle persists in localStorage
   (`gt-theme`).
5. **Old UI replaced outright.** Debug/tone panels are RELOCATED into a console
   drawer, never deleted. Git history is the safety net.

### Non-goals (this pass)
- "Browse all lessons →" screen (the picker lists the existing lesson catalog only;
  the browse entry renders disabled with a `coming soon` title).
- Conversational coach backend (OBS-001 re-scope is its own project) — the coach
  column wires the EXISTING `coachAnswer` flow into the new chrome.
- Light-palette perfection — ship a coherent derivation; owner iterates.
- Mobile/responsive below ~1100 px (desktop practice tool; don't break, don't polish).
- Session-log UI, annotation tool, backend.

## 2. Current state (what's being untangled)

`App.tsx` renders `<h1>` + theme toggle + `<SetupWizard/>` + `<CoachPanel/>`.
**`SetupWizard.tsx` is currently the whole app**: device selects, start/stop, video +
`OverlayCanvas`, calibration mode (manual 4-tap + reset), `InputMeter`,
`OpenStringCheck`, latency probe + advice, `TonePanel`, mode toggle +
`LessonPanel`/`ExplorePanel`, `DebugPanel`, `AudioDebugPanel`. The v2 port splits this
into a one-time **Wizard** flow and a persistent **PracticeScreen** console. Capture
lifecycle (`startCapture`/`CaptureHandles`) and every store stay UNCHANGED — this is a
presentation reorganization; `controller.ts` is read-only for the whole project.

## 4. Design tokens — `apps/web/src/theme/tokens.css`

Semantic CSS custom properties on `:root` (LIGHT values) and `[data-theme="dark"]`
(DARK = the prototype look). Starting values below were sampled from prototype
screenshots — they are STARTING points, tune-by-eye welcome, but all usage goes
through the token names. **Hard rule: no new hardcoded colors outside tokens.css**
(existing App.css hexes migrate as their rules are touched; a grep gate in §11
enforces no NEW ones).

Dark (prototype): `--bg #171310`, `--surface #201a16`, `--surface-raised #282019`,
`--border #352b22`, `--text #efe5d8`, `--text-muted #8d8175`, `--accent #e6a95c`
(amber; buttons/active dots), `--accent-ink #241a10` (text on accent), `--ok #86c78f`,
`--warn #e0b13e`, `--danger #d9776b`, `--info #9db4c9`.
Light (derived): `--bg #faf6f0`, `--surface #ffffff`, `--surface-raised #f3ece2`,
`--border #e2d7c8`, `--text #2a241d`, `--text-muted #7d7265`, same accent family
darkened for contrast (`--accent #c07f2e`), status colors darkened one step. WCAG AA
for text-on-bg in both themes is a review checkpoint, not an afterthought.
Also: `--radius-sm 6px / --radius-md 10px / --radius-lg 16px`, spacing scale
`--sp-1..--sp-6` (4/8/12/16/24/40), `--font-ui` (Hanken Grotesk, already shipped),
`--font-mono` (IBM Plex Mono, already shipped — used for ALL metadata/telemetry/labels
per the prototype's mono-caps idiom), `--pane-gap`.
Theme mechanics: keep `data-theme` attribute; App sets `dark` on first run, reads
`localStorage.gt-theme` after; `statusPalette.ts`/`statusColors.ts` (canvas colors)
gain a theme-aware read so overlay/strip colors match the active theme — additive
change, existing exported names stay.

ConsoleDrawer portal and the backtick key handler.

**`TopBar.tsx`** — left→right per prototype: lesson picker dropdown (existing lesson

via the existing handles path); input badge (`DI · Scarlett 2i2` style — label from

**`PracticeScreen.tsx`** — CSS grid: `[camera+zoom+hint | coach-column(320px)]`,
camera pane dominant (prototype proportions). Camera pane wraps the existing
video/OverlayCanvas mount plus two NEW overlay chips (pure presentation, reading
existing stores): TARGET card bottom-left (chord name + spelling `x 3 2 0 1 0` derived
from the active lesson step fingering, or explore chord/voicing when exploring) and
per-string status chips bottom-right (same data as today's chips). `calibrate` ghost
button top-right.

**`HintBar.tsx`** — the big hint sentence (from `fusionHot`/snapshot as today) +
mono sub-line (`ok · conf 91% · hint p50 61 ms · fused audio+vision` — existing
telemetry fields) + `tip was wrong` button (existing `flagTipWrong`).

**`CoachColumn.tsx`** — practice mode: COACH header, `local-only` badge-toggle,
`Last: …` line, latest coach answer + source label, question textarea + `Ask coach`
(all existing CoachPanel logic, restyled — extract the logic into a hook
`useCoach()` if the panel's current structure resists restyling; behavior unchanged).
Explore mode: renders the ExplorePanel CONTROLS (pickers/pager/tier/no-voicings —
same components/testids), while the strip renders in the ZoomPane slot.

**`TelemetryFooter.tsx`** — one mono line composed from existing observability data
(the prototype string: `webgpu · rVFC · glass→worker … · vision N fr · ring N rd ·
drop 0 · diag N · hints N · in -21 dB / floor −62 · tone AMP 5.8 ms`) — every field
maps to an existing store/histogram value; fields whose source doesn't exist yet are
OMITTED (no fake data). Right side: `setup wizard` (clears `gt-setup-done` → Wizard)

**`ConsoleDrawer.tsx`** — slide-up panel over the footer, sections: Audio

Sections are the EXISTING components mounted as-is (their internal styling gets token

## 6. ZoomPane — `apps/web/src/shell/ZoomPane.tsx` (the new capability)

Hybrid per decision 2. Layout slot: full-width strip under the camera pane, header
`FRETBOARD ZOOM — LIVE OVERLAY`, fixed aspect ≈ 720×180 content box.

**Calibrated path (live crop):**
1. Each rVFC frame (hook into the existing `videoFrameLoop`/overlay draw cycle — do
   NOT add a second rVFC loop; extend the existing callback), compute the fretboard
   quad in IMAGE space: corners = `applyHomography(Hinv, {x,y})` for
   (x,y) ∈ {(fretLineX(0), stringY(1)), (fretLineX(MAX_FRET), stringY(1)),
   (fretLineX(0), stringY(6)), (fretLineX(MAX_FRET), stringY(6))} — the same `Hinv`
   the dot projection already uses.
2. Padded axis-aligned bbox of the quad (pad = 12% of bbox height), clamped to
   `[0, videoWidth]×[0, videoHeight]` in intrinsic video pixels.
3. `zoomCtx.drawImage(video, sx, sy, sw, sh, 0, 0, canvasW, canvasH)` — one draw per
   frame, full-res source (sharper than CSS scaling; research doc §zoom).
4. Re-project dots: for each `TargetDot` already computed for the main overlay
   (lesson `planTargets` or `exploreDots` — reuse, never recompute geometry), map
   `zx = (X/w*videoWidth - sx)/sw * canvasW`, same for y; draw with the same dot
   renderer style (filled/hollow/labels) at zoom scale.
5. Per-string ✓/– ticks on the right edge (same data as camera chips) and the
   left-edge open/mute markers, matching the prototype.
6. Confidence gating identical to the main overlay (ADR-007): calibration lost →
   crossfade to fallback within ~500 ms (reuse `decayConfidence`).

**Fallback path:** no calibration, camera off, or capture not running → render
`<FretboardStrip target={…} heard={…}/>` (lesson target converted to a display
voicing where sensible: lesson `Fingering` → per-string array via existing
`FusionTarget.expectedStrings`/`avoidStrings`; explore target passed as-is). The pane
NEVER goes blank: worst case it shows the empty schematic board.

**Perf budget:** the added work is one `drawImage` + ≤ ~30 dot transforms per frame on
the main thread. Measure with the existing latency histogram discipline: if frame
budget regresses (footer `rVFC`/glass→worker numbers), drop the zoom redraw to every
2nd frame before shipping (decision recorded in code comment).

**Testing:** pure helpers extracted and unit-tested — `zoomCropRect(H, videoW, videoH,
pad)` (quad→padded-clamped bbox; fixtures reuse targetDots' homography fixtures) and
`toZoomSpace(dot, crop, canvasSize)` (linear map, exact expectations). Canvas painting
itself is covered by e2e assertion that the pane's canvas is non-blank when the e2e
calibration fixture is applied, plus the fallback assertion (uncalibrated → strip
testid visible).

## 7. Wizard v2 — reskin of the 3-step flow

Same logic, new presentation (centered card, title `Guitar tutor`, mono subtitle
`one-time setup · under a minute`, 3 progress dots, `skip setup for now` ghost link):
- **Step 1 Camera & input:** intro copy (write fresh in the prototype's voice — do
  not copy its text verbatim if it misstates behavior), camera + audio-input selects
  (existing), `direct input` badge when `classifyAudioInput` = interface,
  `Start capture` (amber) → on running shows the two preview panes: full scene
  (live video) and fretboard zoom (the SAME ZoomPane component in preview size —
  reuse, not a copy; uncalibrated at this point it shows the schematic fallback with
  a `calibrate on the practice screen` hint) + `Capturing ✓` state; `Continue →`.
- **Step 2 Signal check:** copy per prototype; INPUT meter (existing InputMeter,
  token-styled) with `-N dB · floor −62 dB` mono readout; open-string chips
  (existing OpenStringCheck, restyled chips, `n/6` counter); `Measure round-trip`
  + the latencyAdvice line (existing).
- **Step 3 You're set:** ✓ summary lines composed from real state (camera label,
  calibration state, input label + kind + level, open strings n/6 + round trip ms);
  `Start practicing` → sets `gt-setup-done`, mounts PracticeScreen (capture keeps
  running across the transition — the wizard must NOT stop/restart capture on exit).
Wizard state machine stays in a single `wizard/Wizard.tsx` + step components;
capture/device logic remains in captureStore/controller exactly as today.

## 8. Explore & lessons in the new shell

- Mode toggle behavior byte-identical (stopLesson on entering explore, hot-target
  clear on leaving) — it only MOVES to the TopBar.
- Explore's strip renders in the ZoomPane slot: uncalibrated → schematic (today's
  behavior); calibrated → live crop with explore dots re-projected (new win).
- Lesson picker consumes the existing lesson catalog; starting/stopping a lesson and
  step navigation call the existing fusionStore functions; no fusion changes.

## 9. Error / edge / empty states (all designed, none accidental)

- Capture not running on PracticeScreen (skipped wizard): camera pane shows a
  `start capture` card (device selects + start button inline); everything else renders.
- Camera permission denied / device error: existing error strings surface in the
  camera pane card, token-styled, with retry.
- Calibration absent: camera pane shows the existing nudge; ZoomPane falls back.
- Coach backend unreachable / local-only: existing CoachPanel behavior, restyled
  (local-only badge state is the existing toggle).
- Empty lesson picker (no lessons): picker shows `no lessons installed` disabled row.
- Drawer open + wizard rerun simultaneously: entering the wizard closes the drawer.
- Theme with no saved preference: dark (prototype default).
